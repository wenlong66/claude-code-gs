### [会话：使用 /design-system 编写 GDD](session-design-system-skill.md)
**类型：** 设计（技能驱动）
**技能：** `/design-system`
**时长：** 约 60 分钟（14 轮）
**复杂度：** 中等

**场景：**
开发者在 `/map-systems` 生成 systems index 后运行 `/design-system movement`。技能会从游戏概念和依赖 GDD 中读取上下文，先做技术可行性预检，然后逐个引导完成全部 8 个 GDD 部分——每一节都是草拟、批准、写盘后再进入下一节。

**关键时刻：**
- 技术可行性预检标记出 Jolt 物理默认变更（Godot 4.6）
- 增量写入：每一节在批准后立即落盘
- 在第 5 节期间会话崩溃 → 智能体从第一个空白节恢复
- 依赖信号（stamina、inventory）在 Dependencies 部分中被暴露出来
- 结尾明确交接：先运行 `/design-review` 再做下一个系统

**学习要点：**
- `/design-system` 与“让智能体直接写一个 GDD”有什么不同
- 分节循环如何避免 3 万 token 的上下文膨胀
- 增量写文件如何在会话崩溃后仍然有效
- 技能如何暴露下游依赖契约

### [会话：完整故事生命周期](session-story-lifecycle.md)
**类型：** 完整工作流
**技能：** `/story-readiness` → 实现 → `/story-done`
**时长：** 约 50 分钟（13 轮）
**复杂度：** 中等

**场景：**
开发者从冲刺待办中接手一个故事。`/story-readiness` 在任何代码写入前就抓到了一个滚动方向歧义。实现后，`/story-done` 验证了 9 条验收标准，识别出 2 条延期标准（inventory 还未集成），并带着记录关闭了故事。

**关键时刻：**
- `/story-readiness` 在第 2 轮就抓到规范歧义——在实现开始前解决
- ADR 状态检查：如果 ADR 仍是 Proposed，故事会被 BLOCKED
- Manifest version 检查：确认故事的指导没有偏离当前架构
- 延期标准被记录下来（不会丢失）
- `sprint-status.yaml` 在故事关闭时更新，自动显示下一个就绪故事

**学习要点：**
- 为什么 `/story-readiness` 可以避免实现后期才发现歧义
- 延期标准如何工作（COMPLETE WITH NOTES vs. BLOCKED）
- TR-ID 引用如何防止误报偏差
- 从待办 → 已实现 → 已关闭的完整闭环

### [会话：阶段门禁与阶段转换](session-gate-check-phase-transition.md)
**类型：** 阶段门禁
**技能：** `/gate-check`
**时长：** 约 20 分钟（7 轮）
**复杂度：** 低

**场景：**
开发者完成系统设计阶段，并运行 `/gate-check` 以推进阶段。门禁发现所有 6 个 MVP GDD 都已完成，并且跨审查通过，只剩一个低严重度的提示。门禁通过，`stage.txt` 更新，智能体还提供了一份具体、按顺序排列的下一阶段检查清单。

**关键时刻：**
- 门禁不仅验证文件存在，还验证内部完整性（每个 GDD 8 个部分）
- CONCERNS 不等于 FAIL：低严重度跨审查备注仍可通过门禁
- `stage.txt` 更新后，后续 `/help`、`/sprint-status` 和所有技能看到的阶段都会改变
- 智能体把跨审查提示转换成下一步要写的具体 ADR
- 下一阶段清单具体且有顺序，不是泛泛而谈

**学习要点：**
- 门禁到底验证什么（不只是“文件在不在”）
- PASS / CONCERNS / FAIL 如何工作
- 为什么 `stage.txt` 是阶段追踪的权威来源
- 阶段转换后会发生什么

### [会话：UX 流程 — /ux-design → /ux-review → /team-ui](session-ux-pipeline.md)
**类型：** UX 设计流程
**技能：** `/ux-design`、`/ux-review`、`/team-ui`
**时长：** 约 90 分钟（16 轮）
**复杂度：** 中-高

**场景：**
开发者设计 HUD 和库存界面。`/ux-design` 读取玩家旅程和 GDD，让决策建立在玩家情绪状态之上。`/ux-review` 抓出一个阻塞性可访问性缺口（没有拖放的键盘替代方案）和一个建议性颜色盲问题。修复后，`/team-ui` 接手通过。

**关键时刻：**
- HUD 哲学选择（diegetic vs. persistent vs. tactical）基于生存类游戏约定
- `/ux-review` 区分 BLOCKING（阻止交接）与 ADVISORY（可在视觉打磨阶段修复）
- 可访问性在设计阶段就被发现，而不是 QA 阶段
- 一轮内加入键盘替代方案；复查后通过
- `/team-ui` 在开始视觉设计前先检查是否已有通过的 `/ux-review`

**学习要点：**
- `/ux-design` 如何使用玩家旅程上下文来支撑 UI 决策
- `/ux-review` 实际检查什么（不只是“有没有规格文档”）
- HUD 文档（`design/ux/hud.md`）与单屏规格的区别
- 可访问性问题如何在设计阶段而不是实现阶段被处理

### [棕地入门：使用 /adopt](session-adopt-brownfield.md)
**类型：** 棕地迁移
**技能：** `/adopt`
**时长：** 约 30 分钟（8 轮）
**复杂度：** 低-中等

**场景：**
开发者已经有 3 个月的现有代码和粗略设计笔记，但格式都不正确。`/adopt` 会审计格式合规性（不只是文件存在性），按严重度分类 4 个缺口，创建一个有顺序的 7 步迁移计划，并立即通过从代码库推断系统索引来修复 BLOCKING 缺口（缺少 systems index）。

**关键时刻：**
- FORMAT 审计区分“文件存在”和“文件内部结构是否满足要求”
- 识别出 BLOCKING 缺口：缺少 systems index 会阻止 4+ 个技能运行
- 迁移计划按顺序排列：先阻塞项，再高、中、低
- systems index 从代码结构中引导生成——棕地代码里本来就有答案
- Retrofit 模式 vs. 新写模式：`/design-system retrofit` 只补缺口，不覆盖已有内容

**学习要点：**
- `/adopt` 与 `/project-stage-detect` 的区别
- 如何检查格式合规性（检查章节，而不是只看文件是否存在）
- 棕地项目如何在不丢失已有工作的情况下接入系统
- 什么时候用 retrofit 模式，什么时候用完整新写

### [会话：设计制作系统](session-design-crafting-system.md)
**类型：** 设计
**智能体：** game-designer
**时长：** 约 45 分钟（12 轮）
**复杂度：** 中等

**场景：**
单人开发者需要设计一个支撑支柱 2（“通过实验实现涌现式发现”）的制作系统。智能体引导他们问答，给出 3 个带游戏理论分析的设计选项，吸收用户修改，并在每一步都获得批准后逐步起草 GDD。

**关键协作时刻：**
- 智能体在开始时提出 5 个澄清问题
- 呈现 3 个不同选项，并附优缺点 + MDA 对齐
- 用户修改推荐选项，智能体立即整合
- 主动标记边缘情况（“如果是非配方组合呢？”）
- 每一节 GDD 在进入下一节前都先展示供批准
- 创建文件前明确问“我可以把它写到 [file] 吗？”

**学习要点：**
- 设计智能体如何询问目标、约束、参考
- 如何用游戏设计理论（MDA、SDT、Bartle）呈现选项
- 如何按章节迭代草稿
- 什么时候把工作委托给专长智能体（systems-designer、economy-designer）

### [实现战斗伤害计算](session-implement-combat-damage.md)
**类型：** 实现
**智能体：** gameplay-programmer
**时长：** 约 30 分钟（10 轮）
**复杂度：** 低-中等

**场景：**
用户已经有完整设计文档，希望实现伤害计算。智能体先读规范，识别 7 处歧义/缺口，提出澄清问题，给出架构供批准，实施时执行规则，并主动编写测试。

**关键协作时刻：**
- 智能体先读设计文档，识别 7 处规范歧义
- 在实现前先用代码样例提出架构
- 用户要求类型安全后，智能体调整并重新提议
- 规则发现问题（硬编码值），智能体透明修复
- 按验证驱动开发主动编写测试
- 智能体给出下一步选项，而不是擅自假设

**学习要点：**
- 实现型智能体如何在编码前澄清规范
- 如何用代码样例呈现架构供批准
- 规则如何自动执行标准
- 如何处理规范缺口（问，不要假设）
- 验证驱动开发（测试证明它有效）

### [范围危机：战略决策制定](session-scope-crisis-decision.md)
**类型：** 战略决策
**智能体：** creative-director
**时长：** 约 25 分钟（8 轮）
**复杂度：** 高

**场景：**
单人开发者面临危机：Alpha 里程碑还有 2 周，制作系统还需要 3 周，而且投资人演示至关重要。创意总监会收集上下文、正确框定决策、用诚实的权衡分析呈现 3 个战略选项、给出推荐但把决定权留给用户，然后用 ADR 和演示脚本记录决策。

**关键协作时刻：**
- 智能体在提出方案前先读上下文文档
- 提出 5 个问题，理解决策约束
- 正确框定问题（赌注是什么，评价标准是什么）
- 用风险分析和历史先例呈现 3 个选项
- 给出强烈推荐，但明确说：“这是你的决定”
- 记录决策 + 提供演示脚本支持用户

**学习要点：**
- 领导型智能体如何框定战略决策
- 如何用权衡分析呈现选项
- 如何在推荐中使用游戏开发先例和理论
- 如何记录决策（ADR）
- 如何把决策传播给受影响的部门

## 🎯 **这些示例展示了什么**

所有示例都遵循**协作工作流模式**：

```text
Question → Options → Decision → Draft → Approval
```

> **注意：** 这些示例以对话文本的形式展示协作模式。
> 实际使用时，智能体现在会在决策点使用 `AskUserQuestion` 工具，呈现结构化的选项选择器（包含标签、描述和多选）。
> 模式是 **Explain → Capture**：智能体先在对话中解释分析，再通过结构化 UI 让用户做决定。

### ✅ **展示的协作行为：**

1. **智能体先问再假设**
   - 设计智能体会问目标、约束、参考
   - 实现智能体会澄清规范歧义
   - 领导智能体会在推荐前收集完整上下文

2. **智能体给出选项，而不是命令**
   - 2-4 个选项，附优缺点
   - 基于理论、先例和项目支柱的理由
   - 会给建议，但决定权在用户

3. **智能体先展示工作，再最终定稿**
   - 设计草案按章节展示
   - 架构提案在实现前展示
   - 战略分析在决策前展示

4. **智能体在写文件前先获得批准**
   - 使用 Write/Edit 工具前明确说“我可以把它写到 [file] 吗？”
   - 多文件更改会先列出所有受影响文件
   - 用户说“可以”之后才创建文件

5. **智能体会根据反馈迭代**
   - 用户修改会被立即吸收
   - 用户改变建议时不会表现出防御性
   - 当用户改进了智能体建议时，要主动认可

## 🔍 **所有示例中的共同模式**

### 第 1-2 轮：**先理解，再行动**
- 智能体读取上下文（设计文档、规范、约束）
- 智能体提出澄清问题
- 不做假设、不猜

### 第 3-5 轮：**给出带理由的选项**
- 2-4 种不同方法
- 每种方法都有优缺点
- 用理论/先例支撑分析
- 给出推荐，但把决定权留给用户

### 第 6-8 轮：**迭代草案**
- 分阶段展示工作
- 立即吸收反馈
- 主动标记边缘情况或歧义

### 第 9-10 轮：**批准与完成**
- “我可以把它写到 [filepath] 吗？”
- 用户：“可以”
- 智能体写文件
- 智能体给出下一步（测试、审查、集成）

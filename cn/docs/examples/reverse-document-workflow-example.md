# 逆向文档工作流程示例

## 场景
开发者做出了一个可用的技能树系统，但从来没有写过设计文档。
代码在 `src/gameplay/skills/` 中，但 `design/gdd/` 里没有 `skill-system.md`。

---

## 会话记录

**User：**
```text
我在 src/gameplay/skills/ 里有一个可用的技能树系统，大概 1200 行代码。我从来没写过设计文档。我们能根据实现来创建一份吗？
```

**Game-Designer（读取代码并分析）：**
```text
我已经分析了你的技能树实现。下面是我的发现：

结构：
- 3 个技能树：Combat、Survival、Tech
- 每棵树 5 个层级（通过等级 + 前置条件解锁）
- 总共 45 个技能（每棵树 15 个）
- 成本：技能点（每级 1 点，可花金币重置）

机制：
- 被动技能（属性加成）
- 主动技能（有冷却时间的能力）
- 协同加成（第 4 层及以上的跨树解锁）

实现细节：
- 数据驱动（`assets/data/skills/` 下的 JSON 文件）
- 可保存兼容（技能状态会被序列化）
- UI 显示锁定 / 已解锁 / 可用状态

在我起草设计文档前，有几个问题：

1. **设计意图：** 3 棵树是为了代表玩家原型（fighter / survivor / engineer），还是为了提供玩法灵活性？

2. **重置成本：** 重置要花金币——这是为了：
   - 抑制频繁重置（让构筑更有承诺感）？
   - 充当金币消耗口（经济平衡）？
   - 还是应该免费（鼓励尝试）？

3. **协同系统：** 第 4 层出现跨树协同。这是：
   - 预期设计（奖励后期混合构筑）？
   - 还是涌现结果（意外形成混合主流）？

4. **平衡哲学：** 我看到有些技能明显更强（第 5 层被动是 +50% 伤害）。这是：
   - 预期的力量幻想（后期应当很强）？
   - 还是失衡（需要调优）？
```

**User：**
```text
好问题！意图是这样的：

1. 是的，3 棵树是原型，但我想保留灵活性
2. 重置应该很便宜——我希望鼓励实验
3. 协同是故意设计的——混合构筑就是功能的一部分
4. 对，第 5 层应该很强，但 +50% 可能太高了
```

**Game-Designer：**
```text
很好，这就把愿景说明白了。让我起草设计文档：

[展示 design/gdd/skill-system.md 的草稿]

章节：
1. 概述（3 树系统、鼓励混合、鼓励重置）
2. 设计支柱：
   - 原型清晰（每棵树都有身份）
   - 构筑灵活（不锁死、重置便宜）
   - 协同奖励（跨树组合可行）
3. 详细设计（树结构、解锁规则、重置）
4. 平衡框架（层级强度曲线、协同限制）
5. 边缘情况（战斗中重置？技能数上限？钳制？）
6. 依赖（UI、存档系统、教程）
7. 验收标准（所有树都可玩、10+ 种混合构筑、提示清晰）
8. 待解问题：
   - 重置成本应随等级增长还是保持固定？
   - 第 5 层伤害加成：调到 +30% 还是保留 +50%？

我可以把这份内容写到 `design/gdd/skill-system.md` 吗？
```

**User：**
```text
可以写。但我们把第 5 层调到 +30%，而且让重置成本随等级增长。
```

**Game-Designer：**
```text
✅ 已写入 `design/gdd/skill-system.md`
✅ 已标记为 [REVERSE-DOCUMENTED FROM IMPLEMENTATION]
✅ 已添加 TODO：把第 5 层被动改为 +30% in `assets/data/skills/combat-tree.json`
✅ 已添加 TODO：实现阶梯式重置成本公式

下一步：
- 运行 /balance-check 来验证技能树进度曲线？
- 创建 ADR 说明为什么是 3 棵树而不是 5 棵？
- 更新教程，解释混合协同？
```

---

## 关键观察

1. **智能体先问澄清问题** —— 没有直接描述代码
2. **把意图和实现分开** —— 用户纠正了“偶然”的失衡
3. **设计文档记录“为什么”** —— 不只是“代码做了什么”
4. **识别缺失内容** —— 代码里没有的边缘情况
5. **标出了后续工作** —— 平衡调优、ADR、教程

## 结果

设计文档现在已经存在，并且**与现实一致**，同时也记录了**预期愿景**。未来的改动会引用这份文档，新团队成员也能理解这些决策背后的*原因*。

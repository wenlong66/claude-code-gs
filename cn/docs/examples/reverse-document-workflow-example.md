# 逆向文档工作流程示例

## 场景
开发者构建了一个工作的技能树系统但从未编写设计文档。代码存在于 `src/gameplay/skills/`，但 `design/gdd/` 没有 skill-system.md。

---

## 会话记录

**用户：**
```
我在 src/gameplay/skills/ 有一个工作的技能树系统，大约 1200 行代码。
我从未编写设计文档。我们可以从实现中创建一个吗？
```

**游戏设计师（阅读代码，分析）：**
```
我已经分析了你的技能树实现。以下是我发现的：

结构：
- 3 个技能树：战斗、生存、技术
- 每个树 5 层（通过等级 + 先决条件解锁）
- 总共 45 个技能（每个树 15 个）
- 成本：技能点（每级 1 点，可以用金币重新分配）

机制：
- 被动技能（属性加成）
- 主动技能（带冷却时间的能力）
- 协同加成（4 层及以上跨树解锁）

实现细节：
- 数据驱动（assets/data/skills/ 中的 JSON 文件）
- 保存兼容（技能状态序列化）
- UI 显示锁定/解锁/可用状态

在我起草设计文档之前的问题：

1. **设计意图：** 3 树结构是为了代表玩家原型（战士/生存者/工程师）还是游戏风格灵活性？

2. **重新分配成本：** 重新分配消耗金币，这旨在：
   - 阻止频繁重新分配（对build的承诺）？
   - 创建金币消耗（经济平衡）？
   - 还是应该免费（鼓励实验）？

3. **协同系统：** 跨树协同出现在 4 层。这是为了：
   - 有意的（后期奖励混合build）？
   - 还是紧急的（意外创造混合meta）？

4. **平衡哲学：** 我看到一些技能明显更好（5 层被动是 +50% 伤害）。这是：
   - 力量幻想（后期应该感觉超模）？
   - 还是不平衡（需要调整）？
```

**用户：**
```
好问题！这是意图：

1. 是的，3 个树是原型但我希望有灵活性
2. 重新分配应该便宜——我希望实验
3. 协同是有意的——混合build是一个功能
4. 是的，5 层很强，但 +50% 可能太多了
```

**游戏设计师：**
```
完美，这澄清了愿景。让我起草设计文档：

[显示 design/gdd/skill-system.md 的草案]

部分：
1. 概述（3 树系统，混合友好，鼓励重新分配）
2. 设计支柱：
   - 原型清晰度（每个树有身份）
   - Build 灵活性（无锁定，便宜的重新分配）
   - 协同奖励（跨树组合可行）
3. 详细设计（树结构、解锁规则、重新分配）
4. 平衡框架（层力量曲线、协同限制）
5. 边缘情况（战斗中重新分配？最大技能？上限？）
6. 依赖关系（UI、保存系统、教程）
7. 验收标准（所有树都可行、10+ 混合build、清晰的工具提示）
8. 开放问题：
   - 重新分配成本应该随等级增加还是保持不变？
   - 5 层伤害加成：调整到 +30% 还是保持 +50%？

**我可以写到 design/gdd/skill-system.md 吗？**
```

**用户：**
```
可以，写吧。但让我们把 5 层调整到 +30%，并让重新分配成本随等级增加。
```

**游戏设计师：**
```
✅ 写到 design/gdd/sill-system.md
✅ 标记为 [从实现逆向文档]
✅ 添加待办：在 assets/data/skills/combat-tree.json 中将 5 层被动更新到 +30%
```

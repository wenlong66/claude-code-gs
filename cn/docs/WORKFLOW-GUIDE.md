## 阶段 0: 设置和配置

### 你需要什么

在开始之前，确保你有：

- **Claude Code** 已安装并正常工作
- **Git** 配合 Git Bash（Windows）或标准终端（Mac/Linux）
- **jq**（可选但推荐 -- 如果缺少，钩子会回退到 `grep`）
- **Python 3**（可选 -- 一些钩子使用它进行 JSON 验证）

### 步骤 0.1: 克隆和配置

克隆仓库并在编辑器中打开：

```bash
git clone <repo-url> my-game
cd my-game
```

### 步骤 0.2: 运行 /start（新用户推荐）

如果你对这个项目是新手或者还不知道自己要做什么游戏：

```
/start
```

这个引导式入职会询问你处于什么阶段（没有想法、模糊想法、清晰概念、现有工作）并引导你到正确的阶段。如果你已经有游戏概念和引擎决策，可以跳过这一步。

### 步骤 0.3: 选择你的引擎

在 Claude Code 中运行 `/setup-engine`。这是最重要的配置步骤 -- 它告诉每个智能体你使用什么引擎、语言和工具链：

```bash
/setup-engine godot 4.6
```

或者不带参数运行 `/setup-engine`，根据你游戏的需求（2D/3D、平台、团队规模、语言偏好）获得交互式推荐。

**`/setup-engine` 做什么：**

- 在 `CLAUDE.md` 中固定引擎和版本
- 用命名约定、性能预算和引擎特定默认值填充 `.claude/docs/technical-preferences.md`
- 检测知识差距（引擎版本比 LLM 训练数据更新）并从网络获取当前文档，以便智能体提供正确的 API 建议
- 在 `docs/engine-reference/` 中创建版本固定的参考文档

**为什么这很重要：** 一旦你设置了引擎，系统就知道使用哪个引擎专家智能体。如果你选择 Godot，像 `godot-specialist`、`godot-gdscript-specialist` 和 `godot-shader-specialist` 这样的智能体会成为你的首选专家。Unity 和 Unreal 专家仍然可用但不会作为主要。

> **手动替代：** 如果你愿意，也可以直接编辑 `CLAUDE.md` 中的技术栈部分。

### 步骤 0.3: 验证钩子是否正常工作

启动一个新的 Claude Code 会话。你应该看到 `session-start.sh` 钩子的输出：

```
=== Claude Code Game Studios -- Session Context ===
Branch: main
Recent commits:
  abc1234 Initial commit
===================================
```

如果看到这个，说明钩子工作正常。如果没有，请检查 `.claude/settings.json` 确保钩子路径对你的操作系统是正确的。

### 步骤 0.4: 创建你的目录结构

`CLAUDE.md` 中列出的目录并不都已存在。根据需要创建它们 -- 系统期望这样的布局：

```
src/                  # 游戏源代码
  core/               # 引擎/框架代码
  gameplay/           # 游戏玩法系统
  ai/                 # AI 系统
  networking/         # 多人游戏代码
  ui/                 # UI 代码
  tools/              # 开发工具
assets/               # 游戏资产
  art/                # 精灵、模型、纹理
  audio/              # 音乐、音效
  vfx/                # 粒子效果
  shaders/            # 着色器文件
  data/               # JSON 配置/平衡数据
design/               # 设计文档
  gdd/                # 游戏设计文档
  narrative/          # 故事、背景、对话
  levels/             # 关卡设计文档
  balance/            # 平衡电子表格和数据
docs/                 # 技术文档
  architecture/       # 架构决策记录
  api/                # API 文档
  postmortems/        # 事后分析
tests/                # 测试套件
prototypes/           # 可丢弃原型
production/           # 冲刺计划、里程碑、发布
  sprints/
  milestones/
  releases/
```

> **提示：** 你不需要在第一天就创建所有这些目录。当你到达需要它们的阶段时再创建。重要的是，当你创建它们时要遵循这个结构，因为 **规则系统** 基于文件路径强制执行标准。`src/gameplay/` 中的代码获取游戏玩法规则，`src/ai/` 中的代码获取 AI 规则，依此类推。

## 阶段 1: 构思与概念

### 本阶段发生什么

你从"没有想法"或"模糊想法"转变为结构化的游戏概念文档。这是你弄清楚**你在制作什么**的地方。

> **提示：** 如果你在阶段 0 运行了 `/start` 并选择了路径 A 或 B，你已经在这里了。`/start` 会自动将你路由到 `/brainstorm`。

### 步骤 1.1: 使用 `/brainstorm` 进行头脑风暴

如果你跳过了 `/start`，这是你的起点。运行头脑风暴技能：

```
/brainstorm
```

或者带有类型提示：

```
/brainstorm roguelike deckbuilder
```

**发生什么：** 头脑风暴技能使用专业工作室技术引导你完成协作式 6 阶段构思过程：

智能体："让我们探索游戏概念。什么类型或主题让你感兴趣？（可选，或留空进行开放探索）"

用户："Roguelike 卡组构建"

智能体："我已经生成了 10 个结合 roguelike 和卡组构建机制的概念种子。以下是前三名：

**概念 A：咒语编译器** - 代码主题的卡组构建游戏，其中卡牌是函数
**概念 B：记忆宫殿** - 心理惊悚游戏，其中卡牌是记忆
**概念 C：生态系统构建者** - 自然主题，其中卡牌是物种

你希望我用 MDA 分析来开发哪个概念？"

用户："概念 B 听起来很独特"

智能体：[对概念 B 执行 MDA 分析、玩家动机映射、受众定位、动词优先设计，在每个阶段展示工作并征求反馈]

"这是完整的分析。我应该将其正式化为结构化的概念文档吗？"

用户："是的"

智能体：[创建结构化概念文档，包含顶级创意、目标受众、核心机制和情感目标]

### 步骤 1.2: 记录获胜概念

将头脑风暴输出正式化。使用**游戏概念模板**：

```
请 Claude 使用 .claude/docs/templates/game-concept.md 中的模板创建游戏概念文档
```

此模板包括：
- 电梯推介（一句话）
- 核心幻想（玩家想象自己在做什么）
- MDA 分解
- 目标受众（Bartle 类型、人口统计）
- 核心循环图
- 独特卖点
- 可比较的游戏和差异化

### 步骤 1.3: 定义你的游戏支柱

在进一步之前，锁定你的**游戏支柱** -- 3-5 个不可协商的设计价值，指导每一个决策。使用模板：

```
请 Claude 使用 .claude/docs/templates/game-pillars.md 中的模板创建游戏支柱文档
```

示例支柱：
- **有意义的选择**：每个决定永久改变某些东西
- **紧急讲故事**：系统创造独特的叙事，而不是脚本
- **可访问的深度**：易于上手，难以掌握

还要定义**反支柱** -- 你的游戏有意避免的事情：
- 我们不追求：照片级真实感、竞争性平衡、线性叙事

> **当你对你的概念和支柱感到满意时，你就准备好进入预生产阶段了。**

## 阶段 2: 预生产和设计

### 本阶段发生什么

你创建定义游戏如何工作的所有设计文档。还没有代码编写 -- 这是纯粹的设计和架构。

### 步骤 2.1: 将你的概念分解为系统

在编写单独的 GDD 之前，枚举你的游戏需要的所有系统：

```
/map-systems
```

这创建 `design/gdd/systems-index.md` — 一个主跟踪文档，它：

- 列出你的游戏需要的每个系统（战斗、移动、UI 等）
- 映射系统之间的依赖关系
- 分配优先级层级（MVP、垂直切片、Alpha、完整愿景）
- 确定设计顺序（基础 → 核心 → 功能 → 呈现 → 打磨）

然后按依赖顺序设计每个系统：

```
/map-systems next
```

这选择最高优先级的未设计系统并将其交给 `/design-system`，后者引导你按部分创建其 GDD。每个完成的 GDD 在开始下一个之前都要通过 `/design-review`。

你也可以直接编写特定系统的 GDD：

```
/design-system combat-system
```

### 步骤 2.2: 创建游戏设计文档 (GDD)

对于游戏中的每个主要系统，在 `design/gdd/` 中创建设计文档。使用模板：

```
请 Claude 使用 .claude/docs/templates/game-design-document.md 中的模板创建 GDD
```

**`design-docs` 规则**（`.claude/rules/design-docs.md`）要求每个 GDD 有这 8 个部分：

| # | 部分 | 内容 |
|---|---------|---------------|
| 1 | **概述** | 系统的一句话总结 |
| 2 | **玩家幻想** | 玩家使用此系统时的想象/感受 |
| 3 | **详细规则** | 明确的机械规则 -- 如果两个设计师阅读这个，他们会构建相同的东西 |
| 4 | **公式** | 每个计算，带变量定义和范围 |
| 5 | **边缘情况** | 在奇怪情况下会发生什么？明确解决。 |
| 6 | **依赖关系** | 此系统连接的其他系统（双向） |
| 7 | **调整旋钮** | 设计师可以安全更改的哪些值，带有安全范围 |
| 8 | **验收标准** | 你如何测试这是否有效？具体、可测量。 |

**你可能创建的示例 GDD 文件：**
- `design/gdd/core-loop.md` -- 基本游戏循环
- `design/gdd/combat-system.md` -- 战斗如何工作
- `design/gdd/progression-system.md` -- 玩家如何进步
- `design/gdd/inventory-system.md` -- 物品管理
- `design/gdd/ability-system.md` -- 玩家能力和技能

### 步骤 2.2: 使用游戏设计师智能体处理复杂机制

对于需要深入机械思考的系统，委托给 `game-designer`：

```
请游戏设计师智能体设计一个制作系统，支持紧急配方（玩家发现组合），技能进度与实验相关。系统应该感觉有回报而不会太 grindy。目标受众是探索者和成就者。
```

**发生什么：** `game-designer` 智能体与你协作创建设计：

智能体："我将帮助设计制作系统。首先，一些问题：
- 配方应该通过实验发现还是明确教授？
- 失败的制作尝试应该有多严厉？
- 目标受众 - 主要是探索者还是成就者？"

用户："通过实验发现。中等惩罚 - 浪费材料但获得部分技能 XP。主要目标探索者。"

智能体："基于此，这里有三种机械方法：

**选项 A：纯发现** - 没有提示，完全自由
优点：最大探索奖励
缺点：高挫折风险，可能需要 wiki

**选项 B：提示系统** - 当材料兼容时的视觉提示
优点：引导探索，减少挫折
缺点：较少令人满意的'啊哈'时刻

**选项 C：混合** - 自由实验 + 技能解锁的可选日志提示
优点：支持两种游戏风格，渐进式脚手架
缺点：实现更复杂

我推荐选项 C，因为它尊重探索者同时提供安全网，并且日志提示可以通过技能进度为成就者提供关卡。
哪种方法符合你的愿景？"

用户："选项 C 听起来完美"

智能体：[迭代起草 GDD 部分 - 概述、玩家幻想、详细规则、公式、边缘情况 - 展示每个并纳入反馈]

"我需要成功速率和技能进度曲线的详细公式工作。让我咨询系统设计师获取数学。"

智能体：[咨询后] "完整草案已准备好，公式由系统设计师验证。经济设计师标记了材料成本需要平衡 - 我已经纳入了他们的建议。

我可以将此写入 design/gdd/crafting-system.md 吗？"

用户："是的"

智能体：[写入文件]

### 步骤 2.3: 设计审查

在任何设计文档交给程序员之前，运行审查：

```
/design-review design/gdd/crafting-system.md
```

**发生什么：** 技能根据 8 部分标准检查文档，验证：
- 是否所有部分都存在且完整？
- 公式是否明确且有定义的变量？
- 边缘情况是否明确解决（不是"TBD"）？
- 依赖关系是否列出双向？
- 验收标准是否可测试？

在继续之前修复任何标记的问题。

### 步骤 2.4: 创建叙事设计（如适用）

如果你的游戏有故事、背景或对话，现在是构建它的时候：

1. **世界构建** -- 使用 `world-builder` 定义派系、历史、地理和你的世界规则
2. **故事结构** -- 使用 `narrative-director` 设计故事弧、角色弧和叙事节拍
3. **角色表** -- 对每个主要角色使用 `narrative-character-sheet.md` 模板

`narrative` 规则（`.claude/rules/narrative.md`）强制执行：
- 跨文档的背景一致性
- 规范级别（核心规范 vs 风味 vs 外传）
- 角色声音配置文件
- 世界规则文档

### 步骤 2.5: 技术架构

在编码开始之前，做出关键技术决策：

```
/architecture-decision "NPC AI 的状态机 vs 行为树"
```

**发生什么：** 技能引导你创建架构决策记录：

智能体："我将帮助记录这个架构决策。首先，让我了解上下文。现在是什么驱动这个决策？"

用户："我们需要为 NPC AI 在状态机和行为树之间选择。我们计划有 50+ 敌人类型。"

智能体："考虑到 50+ 敌人类型，这是我的分析：

**选项 A：有限状态机**
优点：简单，可调试，开销小
缺点：复杂行为变得笨拙，逻辑难以重用

**选项 B：行为树**
优点：高度可重用，可扩展性好，可视化调试
缺点：学习曲线更陡峭，开销略大

**选项 C：效用 AI**
优点：紧急行为，动态决策
缺点：更难调试，需要仔细调整

对于 50+ 类型，我推荐行为树 - 可重用性和可扩展性超过学习曲线。

我应该创建记录这个决策的 ADR 吗？"

用户："是的，选择行为树"

智能体：[在 docs/architecture/ 中创建 ADR，包含上下文、决策驱动因素、所有选项及其优缺点、选择的选项及其理由，以及后果]

**对于复杂的架构决策，使用 `technical-director` 智能体：**

```
请技术总监智能体评估我们是否应该为游戏玩法实体使用 ECS 架构或传统组件系统，考虑我们的团队规模（单人）和引擎选择（Godot 4）。
```

智能体："让我分析你的上下文中的 ECS vs 传统组件决策：

[执行详细分析，考虑 Godot 4 的场景系统、单人开发工作流程、学习曲线、调试工具、性能需求]

**推荐：传统 Godot 基于节点的组件**

理由：[详细理由]

我应该创建记录这个决策的 ADR 并提供完整分析吗？"

用户："是的"

智能体：[创建全面的 ADR]

### 步骤 2.6: 里程碑规划

使用模板定义你的项目里程碑：

```
请 Claude 使用 .claude/docs/templates/milestone-definition.md 中的模板创建里程碑定义
```

典型的独立游戏里程碑：
1. **垂直切片** -- 一个关卡，完全可玩，代表性质量
2. **Alpha** -- 所有核心系统功能正常，占位符美术 OK
3. **Beta** -- 内容完整，所有功能都在，开始打磨
4. **发布候选** -- 仅修复缺陷，准备发布
5. **黄金版** -- 发布它

> **当你的所有设计文档都经过审查并且你的架构已决定时，你就准备好开始构建了。**

## 阶段 3: 原型制作和验证

### 本阶段发生什么

在投入全面生产之前，你用可丢弃原型验证有风险或不确定的机制。这节省了数周的浪费生产努力。

### 步骤 3.1: 确定需要原型的内容

不是所有东西都需要原型。当以下情况时进行原型设计：
- 机制新颖，你不确定它是否有趣
- 技术方法有风险，你不确定它是否可行
- 两个设计选项都看起来可行，你需要感受差异

### 步骤 3.2: 运行原型技能

```
/prototype "带动量的抓钩移动"
```

**发生什么：** 技能与你协作设置原型：

智能体："让我们原型化抓钩移动。首先，让我澄清我们正在测试的假设：

- 我们是否测试基于动量的抓钩感觉是否良好？
- 或者测试物理的技术可行性？
- 或者比较不同的控制方案？

这个原型需要回答的核心问题是什么？"

用户："测试动量传递是否感觉满足并创造技能表达"

智能体："明白了。这是我的建议设置：

**假设**：带动量传递的抓钩将感觉满足并通过基于技能的物理操作创造紧急移动选项

**成功标准**：
- 玩家可以流畅地链接 3+ 个抓钩点而不失去动量

# 上下文管理

上下文是 Claude Code 会话中最关键的资源。主动管理它。

## 文件支持的状态（主要策略）

**文件是记忆，而不是对话。** 对话是临时的，会被压缩或丢失。磁盘上的文件在压缩和会话崩溃后仍然存在。

### 会话状态文件

维护 `production/session-state/active.md` 作为活动检查点。在每个重要里程碑后更新：

- 设计部分被批准并写入文件
- 架构决策做出
- 实现里程碑达到
- 获得测试结果

状态文件应包含：当前任务、进度检查清单做出的关键决策、正在处理的文件和开放问题。

### 状态行块（仅限 Production+）

当项目处于 Production、Polish 或 Release 阶段时，在 `active.md` 中包含可由状态行脚本解析的结构化状态块：

```markdown
<!-- STATUS -->
Epic: 战斗系统
Feature: 近战战斗
Task: 实现碰撞箱检测
<!-- /STATUS -->
```

- 所有三个字段（Epic、Feature、Task）都是可选的——只包含适用的
- 切换焦点区域时更新此块
- 状态行将其显示为面包屑：`战斗系统 > 近战战斗 > 碰撞箱`
- 当没有活动工作焦点时，移除或清空该块

在任何中断（压缩、崩溃、`/clear`）后，首先读取状态文件。

### 增量文件写入

创建多部分文档（设计文档、架构文档、lore 条目）时：

1. 立即创建包含骨架的文件（所有章节标题、空正文）
2. 在对话中一次讨论和起草一个部分
3. 批准后立即将每个部分写入文件
4. 每个部分后更新会话状态文件
5. 写完一个部分后，关于该部分的先前讨论可以安全地被压缩——决策已在文件中

这将上下文窗口保持在仅持有*当前*部分的讨论（~3-5k tokens），而不是整个文档的对话历史（~30-50k tokens）。

## 主动压缩

- **主动压缩**在约 60-70% 上下文使用时，而不是在限制时被动压缩
- **使用 `/clear`** 在不相关任务之间，或在 2+ 次失败纠正尝试后
- **自然压缩点：** 将部分写入文件后、提交后、完成任务后、开始新话题前
- **专注压缩：** `/compact 专注 [当前任务] — 第 1-3 部分已写入文件，正在处理第 4 部分`

## 按任务类型的上下文预算

- 轻量级（读取/审查）：约 3k tokens 启动
- 中量级（实现功能）：约 8k tokens
- 重量级（多系统重构）：约 15k tokens

## 子智能体委托

使用子智能体进行研究和平探索以保持主会话干净。子智能体在自己的上下文窗口中运行，只返回摘要：

- **使用子智能体**当跨多个文件调查、探索不熟悉的代码或进行会消耗 >5k tokens 文件读取的研究时
- **使用直接读取**当你确切知道要检查哪 1-2 个文件时
- 子智能体不继承对话历史——在提示中提供完整上下文

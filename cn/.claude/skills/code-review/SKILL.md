当此技能被调用时：

1. **完整阅读目标文件**。

2. **阅读 CLAUDE.md** 了解项目编码标准。

3. **识别系统类别**（引擎、游戏玩法、AI、网络、UI、工具）并应用类别特定标准。

4. **根据编码标准进行评估**：
   - [ ] 公共方法和类有文档注释
   - [ ] 每个方法的圈复杂度低于 10
   - [ ] 没有方法超过 40 行（不包括数据声明）
   - [ ] 依赖被注入（没有用于游戏状态的静态单例）
   - [ ] 配置值从数据文件加载
   - [ ] 系统公开接口（不是具体类依赖）

5. **检查架构 compliance**：
   - [ ] 正确的依赖方向（引擎 <- 游戏玩法，不是反向）
   - [ ] 模块之间没有循环依赖
   - [ ] 适当的层分离（UI 不拥有游戏状态）
   - [ ] 使用事件/信号进行跨系统通信
   - [ ] 与代码库中既定模式一致

6. **检查 SOLID compliance**：
   - [ ] 单一职责：每个类只有一个变更原因
   - [ ] 开闭：可扩展而无需修改
   - [ ] 里氏替换：子类型可替换为基类型
   - [ ] 接口隔离：没有胖接口
   - [ ] 依赖反转：依赖于抽象，而不是具体

7. **检查常见游戏开发问题**：
   - [ ] 帧率独立性（delta 时间使用）
   - [ ] 热路径中没有分配（更新循环）
   - [ ] 适当的 null/空状态处理
   - [ ] 需要时的线程安全
   - [ ] 资源清理（无泄漏）

8. **以下列格式输出审查**：

```
## 代码审查：[文件/系统名称]

### 标准遵守：[X/6 通过]
[列出失败及行号]

### 架构：[干净 / 小问题 / 发现违规]
[列出具体架构问题]

### SOLID：[合规 / 发现问题]
[列出具体违规]

### 游戏特定关注
[列出游戏开发特定问题]

### 积极观察
[做得好的是什么——始终包含此部分]

### 必需更改
[批准前必须修复的项目]

### 建议
[最好有的改进]

### 判定：[批准 / 带建议批准 / 需要更改]
```

# Soak Test

soak test（也叫 endurance test）是一种带有明确观察目标的长时间游玩会话。不同于 smoke check（范围广、关键路径、约 10 分钟）或单特性 playtest（约 30 分钟），soak test 会持续 **30 分钟到数小时**，用来暴露：

- **Memory leaks** —— 只有在场景切换后才会出现的堆内存缓慢增长
- **Performance drift** —— 随时间加重的帧时间退化
- **State accumulation bugs** —— 只有在某个机制重复 N 次后才会出现的问题（inventory full、score overflow、AI state corruption）
- **Fun fatigue** —— 第一局感觉很好，但长时间游玩后变得重复的机制
- **Content exhaustion** —— 玩家耗尽新内容的时点

**这个技能会生成观察协议和分析脚手架——真正游玩的是人。**

**输出：** `production/qa/soak-test-[date]-[duration].md`

**何时运行：**
- Polish 阶段——在 `/gate-check release` 之前
- 修复 memory 或 stability 问题之后（回归 soak）
- 当长时间游玩尚未被正式追踪时

---

## 1. 解析参数

**Duration**（默认：`1h`）：
- `30m` —— 短 soak；适合测试单个机制或场景
- `1h` —— 标准 soak；覆盖大多数常见泄漏类别
- `2h` —— 扩展 soak；建议作为第一次完整 Polish soak
- `4h` —— 深度 soak；长会话设计游戏（RPG、模拟类）必需

**Focus**（默认：`all`）：
- `memory` —— 关注 heap 大小、对象数量、泄漏模式
- `stability` —— 关注 crash/freeze/hang 检测
- `balance` —— 关注 fun fatigue、content exhaustion、难度感知
- `all` —— 以上全部

## 3. 定义观察检查点

根据持续时间生成定时检查点：

**30m soak**：T+0、T+10、T+20、T+30
**1h soak**：T+0、T+15、T+30、T+45、T+60
**2h soak**：T+0、T+20、T+40、T+60、T+80、T+100、T+120
**4h soak**：T+0、T+30、T+60、T+90、T+120、T+180、T+240

每个检查点都记录 Phase 4 中定义的观察项。

---

## 4. 生成 soak test 协议

### Memory / Stability observation items（如果 focus = memory 或 all）

引擎特定监测指导：

**Godot 4：**
- 打开 Debugger → Monitors tab；跟踪 `Memory → Static Memory` 和 `Object Count → Objects`
- 记录：Static Memory（KB）、Object Count、Orphan Nodes 数量
- 报警阈值：在前 15 分钟之后，Memory 增长超过 T+0 的 20%（加载时有一定增长是正常的；持续增长说明泄漏）
- 注意：在 Godot 4.6 中，`Performance.get_monitor(Performance.MEMORY_STATIC)` 返回字节

**Unity：**
- 打开 Memory Profiler（Window → Analysis → Memory Profiler）
- 记录：Total Reserved Memory（MB）、GC Allocated（MB）、每个检查点的 Object Count
- 报警阈值：GC Allocated 在 3+ 个检查点中单调增长

**Unreal Engine：**
- 每个检查点使用 `stat memory` 控制台命令
- 记录：Physical Memory Used（MB）、Physical Memory Available
- 报警阈值：整个 soak 期间 Physical Memory Used 增长超过 50MB

### Stability observation items（如果 focus = stability 或 all）

每个检查点记录：
- [ ] 自上一个检查点以来没有 crash、hang 或 freeze
- [ ] 帧率仍在目标预算内（[target FPS] fps）
- [ ] 音频仍然正常播放（没有不同步或静音）
- [ ] 所有 HUD 元素仍正常渲染
- [ ] 输入响应正常（没有输入丢失或明显卡顿）

### Balance / fatigue observation items（如果 focus = balance 或 all）

每个检查点收集主观观察：
- [ ] 核心机制仍然有回报感（Y/N）
- [ ] 主观难度： [too easy / appropriate / too hard]
- [ ] 自上次检查点以来有没有出现“我见过这个了”的时刻？（novel content exhaustion）
- [ ] 自上次检查点以来有没有出现过沮丧时刻？记录原因。
- [ ] 自上次检查点以来有没有出现过高峰参与时刻？记录原因。

## 协作协议

- **这个技能生成的是协议——由人来执行**——绝不要自动运行 soak test。观察需要人类观察者。
- **持续时间应该与游戏的会话设计相匹配**——一个 5 分钟的游戏不需要 4 小时 soak；一个 city-builder 可能需要。请结合判断，不确定时要询问。
- **第一次 soak 应该使用 `all` focus**——窄 focus（仅 memory）适用于针对特定修复后的回归 soak，不适用于第一次测试
- **写入前先询问**——创建协议文件前始终确认

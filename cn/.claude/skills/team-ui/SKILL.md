当此技能被调用时，通过结构化管线协调 UI 团队。

**决策点：** 在每个阶段转换时，使用 `AskUserQuestion` 向用户呈现子智能体的提议作为可选选项。在对话中写满智能体的完整分析，然后用简洁的标签捕获决策。在进入下一阶段之前，用户必须批准。

## 团队组成
- **ux-designer** — 用户流程、线框图、可访问性、输入处理
- **ui-programmer** — UI 框架、屏幕、小部件、数据绑定、实现
- **art-director** — 视觉风格、布局打磨、与艺术圣经的一致性

## 如何委托

使用 Task 工具将每个团队成员作为子智能体生成：
- `subagent_type: ux-designer` — 用户流程、线框图、可访问性、输入处理
- `subagent_type: ui-programmer` — UI 框架、屏幕、小部件、数据绑定
- `subagent_type: art-director` — 视觉风格、布局打磨、艺术圣经一致性

始终在每个智能体的提示中提供完整上下文（功能需求、现有 UI 模式、平台目标）。在管线允许的地方并行启动独立智能体（例如，阶段 4 审查智能体可以同时运行）。

## 管线

### 阶段 1：UX 设计
委托给 **ux-designer**：
- 定义此功能的用户流程（入口点、状态、出口点）
- 为每个屏幕/状态创建线框图
- 指定交互模式：键盘/鼠标和游戏手柄如何导航？
- 定义可访问性要求：文本大小、对比度、色盲安全
- 识别 UI 需要显示的数据（它读取什么游戏状态？）
- 输出：带线框图和交互图的 UX 规范

### 阶段 2：视觉设计
委托给 **art-director**：
- 根据艺术圣经审查线框图
- 定义视觉处理：颜色、排版、间距、动画
- 指定资源要求（图标、背景、装饰元素）
- 确保与现有 UI 屏幕的一致性
- 输出：带风格备注的视觉设计规范

### 阶段 3：实现
委托给 **ui-programmer**：
- 根据 UX 规范和视觉设计实现 UI
- 确保 UI 绝对不拥有或修改游戏状态——只显示，事件用于操作
- 所有文本通过本地化系统——不要硬编码字符串
- 支持两种输入方式（键盘/鼠标 + 游戏手柄）
- 实现可访问性功能（文本缩放、色盲模式支持）
- 连接到游戏状态的数据绑定
- 输出：已实现的 UI 功能

### 阶段 4：审查（并行）
并行委托：
- **ux-designer**：验证实现与线框图和交互规范匹配。测试仅键盘和仅游戏手柄导航。检查可访问性。
- **art-director**：验证与艺术圣经的视觉一致性。在最小和最大支持的分辨率下检查。

### 阶段 5：打磨
- 处理审查反馈
- 验证动画可跳过并尊重运动偏好
- 确认 UI 声音通过音频事件系统触发
- 在所有支持的分辨率和宽高比下测试

## 输出
涵盖以下内容的摘要报告：UX 规范状态、视觉设计状态、实现状态、可访问性合规性、输入方法支持以及任何待解决的问题。

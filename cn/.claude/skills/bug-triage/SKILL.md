# Bug Triage

此技能会把开放中的 bug backlog 处理成一份已排序、已分配到 sprint 的行动清单。它区分**严重度**（影响有多糟）与**优先级**（需要多快修），识别系统性趋势，并确保关键 bug 不会在 sprint 之间丢失。

**输出：**`production/qa/bug-triage-[date].md`

**何时运行：**
- sprint 开始时——把开放 bug 分配到新 sprint 或 backlog
- `/team-qa` 完成并新 bug 已提交之后
- 当开放 bug 数量超过 10 个时

---

## 1. 解析参数

**模式：**
- `/bug-triage sprint` —— 针对当前 sprint 做 triage；把可修复 bug 分配到 sprint backlog；其余延后
- `/bug-triage full` —— 不考虑 sprint 范围，对所有 bug 做完整 triage
- `/bug-triage trend` —— 仅做趋势分析（不分配）；只读报告
- 无参数 —— 若存在当前 sprint 则运行 sprint 模式，否则运行 full 模式

---

## 2. 加载 bug backlog

### 步骤 2a — 发现 bug 文件

按优先级 Glob bug 报告：
1. `production/qa/bugs/*.md` —— 单个 bug 报告文件（首选格式）
2. `production/qa/bugs.md` —— 单一汇总 bug 日志（fallback）
3. 任何 `production/qa/qa-plan-*.md` 中的 “Bugs Found” 表格（最后手段）

如果没有找到 bug 文件：
> "在 `production/qa/bugs/` 中没有找到 bug 文件。如果 bug 记录在别的位置，请调整 glob pattern。如果当前还没有 bug，那么就没有什么可 triage 的。"

停止并报告。若没有 bug，不要继续。

### 步骤 2b — 加载 sprint 上下文

读取 `production/sprints/` 下最近修改的文件，以了解：
- 当前 sprint 编号 / 名称
- 范围内的 stories（用于分配目标）
- sprint 容量限制（如果有注明）

如果不存在 sprint 文件：记录 “No sprint plan found — assigning to backlog only.”

### 步骤 2c — 加载严重度参考

如果存在，读取 `.claude/docs/coding-standards.md` 了解严重度/优先级定义。若不存在，则使用第 3 步中的标准定义。

---

## 3. 分类每个 bug

对每个 bug 提取或推断：

### 严重度（bug 的影响）

| 严重度 | 定义 |
|----------|-----------|
| **S1 — Critical** | 游戏崩溃、数据丢失、或功能完全失败。到这里为止无法继续。 |
| **S2 — High** | 主要功能坏了，但游戏仍可玩。存在明显错误行为。 |
| **S3 — Medium** | 功能退化，但有绕过方法。轻微错误行为。 |
| **S4 — Low** | 视觉瑕疵、纯美观问题、拼写错误。对玩法无影响。 |

### 优先级（修复紧迫度）

| 优先级 | 定义 |
|----------|-----------|
| **P1 — Fix this sprint** | 阻塞 QA、阻塞发布、或是上个 sprint 的回归 |
| **P2 — Fix soon** | 应在下一个重大里程碑前解决 |
| **P3 — Backlog** | 值得修，但没有活跃阻塞影响 |
| **P4 — Won't fix / Deferred** | 可接受风险，或超出当前产品范围 |

### 分配

在 `sprint` 模式下，对每个 P1/P2 bug：
- 识别它归属哪个 story 或 epic
- 检查当前 sprint 是否还有剩余容量
- 若有容量：分配到 sprint（`Sprint: [current]`）
- 若容量已满：标记为 `Priority overflow — consider pulling from sprint`

在 `full` 模式下：所有 P1 分配到当前 sprint，P2 分配到下个 sprint 估算，P3+ 放入 backlog。

### 偏差检查

标记可能说明**系统性问题**的 bug：
- 同一 sprint 中同一系统出现 3+ 个 bug → “Potential design or implementation quality issue in [system]”
- 同一 story 中出现 2+ 个 S1/S2 bug → “Story may need to be reopened and re-reviewed before shipping”
- bug 提交对象是标记为 Complete 的 story → “Regression in completed story — story should be re-opened in sprint tracking”

## 4. 趋势分析

分类完所有 bug 后，生成趋势指标：

### 数量趋势
- 开放 bug 总数：[N]
- 本 sprint 新增：[N]
- 本 sprint 关闭：[N]
- 净变化：[+N / -N]

### 系统热点
- 哪个系统开放 bug 最多？
- 哪个系统的 S1/S2 比例最高？

### 年龄分析
- 有多少 bug 已经超过 2 个 sprint？
- 是否有未分配（sprint = none）的 S1/S2 bug？

### 回归指标
- 是否有 bug 指向之前已完成的 stories？
- 数量：[N] 个回归 bug（意味着 story 需要重新打开）

## 6. 写入与门控

先在对话中展示报告，然后询问：

"May I write this triage report to `production/qa/bug-triage-[date].md`?"

只有在批准后才写入。

写入后：
- 如果有未分配的 S1 bug："S1 bugs must be assigned before the sprint can be considered healthy. Run `/sprint-status` to see current capacity."
- 如果存在回归 bug："Regressions found — consider re-opening the affected stories in sprint tracking and running `/smoke-check` to re-gate."
- 如果没有 P1 bug："No P1 bugs — build is in good shape for QA hand-off." 结论：**COMPLETE** — triage report written.

如果用户拒绝写入：结论：**BLOCKED** — user declined write.

---

## 协作协议

- **绝不要在未经用户批准前关闭 bug 或标记为 Won't Fix**——把它们作为 P4 候选项展示，并询问："Are these acceptable as Won't Fix?"
- **绝不要把 bug 自动分配到已满容量的 sprint**——标记 overflow，让 sprint owner 决定是否拉入
- **严重度是客观的；优先级是团队决策**——把严重度分类作为建议，而非强制
- **趋势数据仅供参考**——不要仅凭趋势结果阻塞工作；把它们作为观察项呈现

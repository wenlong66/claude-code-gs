description: "运行本地化工作流程：提取字符串、验证本地化就绪性、检查硬编码文本并生成翻译就绪的字符串表。"
argument-hint: "[scan|extract|validate|status]"

allowed-tools: Read, Glob, Grep, Write, Bash
---
当此技能被调用时：

1. **从参数解析子命令**：
   - `scan` — 扫描本地化问题（硬编码字符串、缺失键）
   - `extract` — 提取新字符串并生成/更新字符串表
   - `validate` — 验证现有翻译的完整性和格式
   - `status` — 报告整体本地化状态

2. **对于 `scan`**：
   - 在 `src/` 中搜索硬编码的面向用户字符串：
     - UI 代码中未包装在本地化函数中的字符串字面量
     - 应该参数化的串联字符串
     - 使用位置占位符（`%s`、`%d`）而不是命名占位符（`{playerName}`）的字符串
   - 搜索本地化反模式：
     - 未使用区域感知函数的日期/时间格式
     - 没有区域意识的数字格式
     - 嵌入图像或纹理中的文本（标记资源文件）
     - 假设从左到右文本方向的字符串
   - 报告所有发现，包含文件路径和行号

3. **对于 `extract`**：
   - 扫描所有源文件以获取本地化字符串引用
   - 与 `assets/data/` 中的现有字符串表（如果有）进行比较
   - 为尚无键的字符串生成新条目
   - 建议遵循以下约定的键名：`[category].[subcategory].[description]`
   - 输出要添加到字符串表的新字符串差异

4. **对于 `validate`**：
   - 读取 `assets/data/` 中的所有字符串表文件
   - 检查每个条目：
     - 缺失翻译（键存在但没有特定区域的翻译）
     - 占位符不匹配（源有 `{name}` 但翻译缺失）
     - 字符串长度违规（超过 UI 元素的字符限制）
     - 孤立键（翻译存在但代码中没有引用该键）
   - 按区域和严重性分组报告验证结果

5. **对于 `status`**：
   - 统计本地化字符串总数
   - 每个区域：统计已翻译、未翻译和过时（自翻译以来源已更改）
   - 生成覆盖矩阵：

   ```markdown
   ## 本地化状态
   生成日期：[日期]

   | 区域 | 总数 | 已翻译 | 缺失 | 过时 | 覆盖 |
   |--------|-------|-----------|---------|-------|----------|
   | en (源) | [N] | [N] | 0 | 0 | 100% |
   | [区域] | [N] | [N] | [N] | [N] | [X]% |

   ### 问题
   - 在源代码中发现 [N] 个硬编码字符串
   - [N] 个字符串超过字符限制
   - [N] 个占位符不匹配
   - [N] 个孤立键（可以清理）
   ```

### 规则
- 英语（en）始终是源区域
- 每个字符串表条目必须包含解释上下文的译者注释
- 永远不要直接修改翻译文件——生成差异供审查
- 字符限制必须为每个 UI 元素定义并自动强制执行
- 从一开始就应考虑从右到左（RTL）语言支持，而不是事后添加

当此技能被调用时：

1. **完整阅读目标设计文档**。

2. **阅读 master CLAUDE.md** 了解项目上下文和标准。

3. **阅读目标文档引用或暗示的相关设计文档**（检查 `design/gdd/` 中的相关系统）。

4. **根据设计文档标准清单进行评估**：
   - [ ] 有概述部分（一段式摘要）
   - [ ] 有玩家幻想部分（预期感受）
   - [ ] 有详细规则部分（明确的机制）
   - [ ] 有公式部分（所有数学用变量定义）
   - [ ] 有边缘情况部分（处理的异常情况）
   - [ ] 有依赖部分（列出的其他系统）
   - [ ] 有调优旋钮部分（识别的可配置值）
   - [ ] 有验收标准部分（可测试的成功条件）

5. **检查内部一致性**：
   - 公式产生的值与描述的行为匹配吗？
   - 边缘情况与主要规则矛盾吗？
   - 依赖是双向的吗（其他系统知道这个吗）？

6. **检查可实现性**：
   - 规则足够精确，程序员可以在不猜测的情况下实现吗？
   - 有没有"含糊其辞"的部分缺失细节？
   - 考虑性能影响了吗？

7. **检查跨系统一致性**：
   - 这与现有机制冲突吗？
   - 这会与其他系统产生意想不到的交互吗？
   - 这与游戏既定基调和支柱一致吗？

8. **以下列格式输出审查**：

```
## 设计审查：[文档标题]

### 完整性：[X/8 部分存在]
[列出缺失的部分]

### 一致性问题
[列出任何内部或跨系统矛盾]

### 可实现性关注
[列出任何模糊或无法实现的部分]

### 平衡问题
[列出任何明显的平衡风险]

### 建议
[优先排序的改进列表]

### 判定：[批准 / 需要修订 / 需要重大修订]
```

9. **上下文下一步建议**：
   - 如果被审查的文档是 `game-concept.md` 或 `game-pillars.md`：
     - 检查 `design/gdd/systems-index.md` 是否存在
     - 如果不存在，在建议中添加：
       > "此概念已准备好进行系统分解。运行 `/map-systems` 将其分解为具有依赖关系和优先级的单个系统，然后编写每个系统的 GDD。"
   - 如果文档是单个系统 GDD：
     - 检查系统索引是否引用了此系统
     - 如果判定是批准：建议"将此系统的系统索引状态更新为'已批准'。"
     - 如果判定是需要修订或需要重大修订：建议"将此系统的系统索引状态更新为'审查中'。"
     - 注意：此技能是只读的。用户（或 `/design-system`）必须在系统索引中执行实际的状态更新。

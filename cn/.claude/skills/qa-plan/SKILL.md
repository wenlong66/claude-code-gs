# QA 计划

这个技能会为 sprint、feature 或单个 story 生成结构化 QA 计划。它会读取所有范围内的 story 文件及其引用的 GDD，按测试类型对每个 story 分类，然后输出一份计划，明确告诉开发者：哪些要自动化、哪些要手工验证、smoke test 覆盖什么，以及何时需要引入 playtester。

请在 sprint 开始前运行它，这样团队就能提前知道需要做哪些测试工作。在实现之后才写测试计划，那更像事后复盘，而不是计划。

**输出：** `production/qa/qa-plan-[sprint-slug]-[date].md`

---

## 第 1 阶段：解析范围

**参数：** `$ARGUMENTS`（空 = 通过 AskUserQuestion 询问用户）

根据参数确定范围：

- **`sprint`** —— 读取 `production/sprints/` 中最近的文件，提取其中引用的所有 story 文件路径。如果存在 `production/sprint-status.yaml`，则把它作为首要 story 列表，并回退使用 sprint plan 获取 story 元数据。
- **`feature: [system-name]`** —— glob `production/epics/*/story-*.md`，筛选出文件路径或标题中包含该 system 名称的 stories。还要检查该系统目录下的 epic 索引文件（`EPIC.md`）。
- **`story: [path]`** —— 验证路径存在并只加载这一份文件。
- **无参数** —— 使用 `AskUserQuestion`：
  - "What is the scope for this QA plan?"
  - 选项："Current sprint"、"Specific feature (enter system name)"、"Specific story (enter path)"、"Full epic"

范围解析后，报告："Building QA plan for [N] stories in [scope]."

如果引用了 story 文件路径但文件不存在，标记为 MISSING 并继续处理其余 stories。不要因为一个缺失文件就让整个计划失败。

---

## 第 2 阶段：加载输入

对每个范围内的 story 文件，完整读取并提取：

- **Story title** 和 story ID（来自文件名或 header）
- **Story Type** 字段（如果文件 header 中存在——例如 `Type: Logic`）
- **Acceptance criteria** —— 完整的编号/项目列表
- **Implementation files** —— 列在 “Files to Create / Modify” 或类似章节下的内容
- **Engine notes** —— 任何引擎 API 警告或版本相关备注
- **GDD reference** —— 引用的 GDD 路径
- **ADR reference** —— 引用的 ADR
- **Estimate** —— 如果有，小时数或 story points
- **Dependencies** —— 该 story 依赖的其他 stories

读完 stories 后，一次性加载支撑上下文（不是逐个 story 加载）：

- `design/gdd/systems-index.md` —— 了解系统优先级，以及哪些 GDD 已批准
- 对所有 story 中引用到的每个唯一 GDD：只读取 **Acceptance Criteria** 和 **Formulas** 章节。不要加载完整 GDD——这两个章节包含可测试需求和要验证的数学公式。
- `docs/architecture/control-manifest.md` —— 如果存在，扫描自动化测试应防护的禁止模式

如果某个 story 没有引用 GDD，记作 gap，但不要因此阻塞计划。该 story 仍可仅基于 acceptance criteria 进行分类。

---

## 第 3 阶段：对每个 story 分类

对每个 story 指定 Story Type。如果 story header 已有 `Type:` 字段，使用其值并按下面标准验证。如果字段缺失或含糊，则根据 acceptance criteria 推断类型。

| Story Type | 分类指标 |
|---|---|
| **Logic** | 验收标准引用计算、公式、数值阈值、状态转移、AI 决策、数据验证、buff/debuff 叠加、经济交易，或任何可测试的计算 |
| **Integration** | 标准涉及两个或多个系统交互、signals/events 跨系统边界传播、save/load 往返、网络同步或持久化 |
| **Visual/Feel** | 标准引用动画行为、VFX、shader 输出、“感觉响应迅速”、感知时序、screen shake、粒子效果、音频同步或视觉反馈质量 |
| **UI** | 标准引用菜单、HUD 元素、按钮、屏幕、对话框、背包面板、tooltip，或任何面向玩家的界面元素 |
| **Config/Data** | 变更仅限于数值调优、数据文件或配置——不涉及新的代码逻辑 |

**混合 story**（例如既包含公式又包含 UI 展示的 story）：按实施风险最高的那部分作为主类型，并注明次类型。Logic+Integration 或 Visual+UI 是最常见的混合组合。

所有 stories 分类完成后，在继续 Phase 4 之前，先在对话中输出一张分类汇总表。这样用户可以清楚地看到测试将如何分配。

---

## 第 4 阶段：生成测试计划

组装完整 QA 计划文档。使用以下结构：

````markdown
# QA Plan: [Sprint/Feature Name]
**Date**: [date]
**Generated by**: /qa-plan
**Scope**: [N stories across [N systems]]
**Engine**: [engine name from .claude/docs/technical-preferences.md, or "Not configured"]
**Sprint File**: [path to sprint plan if applicable]

---

| [story title] | Integration | Integration test — `tests/integration/[system]/` | Smoke check |

| [story title] | UI | Interaction walkthrough | Manual step-through |
| [story title] | Config/Data | Data validation test | Spot-check in-game values |

## Smoke Test Scope

在任何 QA hand-off 之前，本 sprint 需要验证的关键路径：

1. Game launches to main menu without crash
2. New game / new session can be started
3. [Primary mechanic introduced or changed this sprint]
4. [Any system with a regression risk from this sprint's changes]
5. Save / load cycle completes without data loss (if save system exists)
6. Performance is within budget on target hardware (no new frame spikes)

*Smoke tests are verified by the developer via `/smoke-check`. Reference this
list when running that skill.*

---

## Playtest Requirements

| Story | Playtest Goal | Min Sessions | Target Player Type |
|-------|--------------|--------------|-------------------|
| [story] | [What question must the session answer?] | [N] | [new player / experienced] |

**Sign-off requirement**: Playtest notes must be written to
`production/session-logs/playtest-[sprint]-[story-slug].md` and reviewed by
the [designer / qa-lead] before the story can be marked COMPLETE.

如果没有 story 需要 playtest 验证：*本 sprint 不需要 playtest sessions。*

---

## Definition of Done — This Sprint

一个 story 只有在以下全部满足时才算 DONE：

- [ ] All acceptance criteria verified — via automated test result OR documented
      manual evidence (screenshot, video, or playtest notes with sign-off)
- [ ] Test file exists at the specified path for all Logic and Integration stories
- [ ] Manual evidence document exists for all Visual/Feel and UI stories
- [ ] Smoke check passes (run `/smoke-check sprint` before QA hand-off)
- [ ] No regressions introduced
- [ ] Code reviewed (via `/code-review` or documented peer review)
- [ ] Story file updated to `Status: Complete` (via `/story-done`)
````

生成内容时，使用 Phase 2 中提取到的真实 story 标题、GDD formula 文本和 acceptance criteria。不要使用占位文本——每个测试条目都应反映这些特定 stories 的真实需求。

---

## 第 5 阶段：写出输出

在对话中展示完整计划（如果很长，则展示摘要），然后询问：

"May I write this QA plan to `production/qa/qa-plan-[sprint-slug]-[date].md`?"

按生成结果原样写入——不要截断。

写入后：

"QA plan written to `production/qa/qa-plan-[sprint-slug]-[date].md`.

Next steps:
- 在 sprint 实现开始前把这份计划分享给团队
- 所有 stories 实现完毕后运行 `/smoke-check sprint`，作为 QA hand-off 的门控
- 对于 Logic/Integration stories，在标注故事完成前，先创建列出的测试文件路径——`/story-done` 会检查它们"

---

## 协作协议

- **不要在未询问前写入计划**——Phase 5 要求显式批准。
- **保守分类**：当一个 story 在 Logic 和 Integration 之间含糊时，将其归为 Integration——它需要 unit 和 integration 两类测试。
- **不要发明测试用例**：超出 acceptance criteria 和 GDD formulas 支持范围的内容不要自己补。若 GDD 中没有公式，就标记出来，不要猜。
- **Playtest 要求是建议性质**：对于边界上的 Visual/Feel stories，是否需要 playtest 由用户决定。标记该情况即可，不要强制。
- 当没有参数时，使用 `AskUserQuestion` 选择范围。除范围选择外，其他阶段都保持非交互式——先展示结果，再询问一次是否批准写入。

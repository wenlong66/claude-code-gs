当此技能被调用时：

1. **从参数中读取概念描述**。确定此原型必须回答的核心问题。如果概念模糊，在继续之前明确说明问题。

2. **阅读 CLAUDE.md** 了解项目上下文和当前技术栈。了解使用了什么引擎、语言和框架，以便使用兼容的工具构建原型。

3. **创建原型计划**：用 3-5 个要点定义最小可行原型。核心问题是什么？回答它绝对需要的最少代码是什么？什么可以跳过？

4. **创建原型目录**：`prototypes/[concept-name]/`，其中 `[concept-name]` 是从概念派生的简短、kebab-case 标识符。

5. **在隔离目录中实现原型**。每个文件必须以以下内容开头：
   ```
   // PROTOTYPE - NOT FOR PRODUCTION
   // Question: [正在测试的核心问题]
   // Date: [当前日期]
   ```
   标准有意放宽：
   - 自由硬编码值
   - 使用占位符资源
   - 跳过错误处理
   - 使用最简单的可行方法
   - 复制代码而不是从生产导入

6. **测试概念**：运行原型。观察行为。收集任何可测量数据（帧时间、交互次数、感觉评估）。

7. **生成原型报告** 并保存到 `prototypes/[concept-name]/REPORT.md`：

```markdown
## 原型报告：[概念名称]

### 假设
[我们期望什么是真的——我们试图回答的问题]

### 方法
[我们构建了什么，花了多长时间，我们走了什么捷径]

### 结果
[实际发生了什么——具体观察，不是意见]

### 指标
[测试期间收集的任何可测量数据]
- 帧时间：[如果相关]
- 感觉评估：[主观但具体——"200ms 延迟时响应感觉迟缓"不是"感觉不好"]
- 玩家动作次数：[如果相关]
- 迭代次数：[多少次尝试使其工作]

### 建议：[继续 / 转向 / 终止]

[一段解释带有证据的建议]

### 如果继续
[生产质量实现需要什么更改]
- 架构要求
- 性能目标
- 原始设计的范围调整
- 预计生产工作量

### 如果转向
[结果建议的替代方向]

### 如果终止
[为什么这个概念不起作用，我们应该做什么]

### 经验教训
[影响其他系统或未来工作的发现]
```

8. **向用户输出摘要**：包含核心问题、结果和建议。链接到完整报告在 `prototypes/[concept-name]/REPORT.md`。

### 重要约束

- 原型代码绝对不能从生产源文件导入
- 生产代码绝对不能从原型目录导入
- 如果建议是继续，生产实现必须从头编写——原型代码不会迁移到生产
- 总原型工作应该时间框为 1-3 天的等效工作
- 如果原型范围开始增长，停止并重新评估问题是否可以简化
